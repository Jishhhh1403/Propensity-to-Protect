Cloud Propensity Pipeline
End-to-end guide in plain language

This guide explains the full workflow like you are new to engineering. Follow the steps exactly in order.

1) What this system does

Think of this like a factory:

  1. It reads raw files from cloud storage (GCS).
  2. It cleans and organizes the data.
  3. It creates useful behavior features.
  4. It stores results in BigQuery tables.
  5. It can build 30-day "before purchase" sequences for policy events.
  6. It can train a GRU model and write model outputs back to BigQuery.

2) Main output tables

TableWhat it contains
daily_featuresOne row per customer per day with engineered features.
policy_eventsPolicy event rows (purchase date, policy details, risk fields).
policy_2024_2025_30_day_sequences30-day windows before policy purchase dates.
policy_propensity_gru_embeddingsGRU model outputs: embeddings and probabilities.

3) One-time setup

A. Create environment

cd "C:\Users\jishn\Development\cloud-propensity-pipeline"
py -m venv .venv
.\.venv\Scripts\Activate.ps1
pip install -r requirements.txt

B. Configure .env

Copy .env.example to .env and set at least:

  • GCP_PROJECT_ID
  • GCS_BUCKET, GCS_PREFIX
  • BQ_DATASET and table names
  • VERTEX_IMAGE_URI (for Vertex runs)

C. Google login (ADC)

gcloud auth application-default login
gcloud auth application-default set-quota-project <YOUR_PROJECT_ID>

4) Run options (pick one)

Option 1: Fast local test

.\.venv\Scripts\python.exe -m src.pipelines.run_all_pipeline

Good for quick sanity checks (small sampled set).

Option 2: Main local pipeline for 2024/2025

.\.venv\Scripts\python.exe -m src.pipelines.run_policy_2024_2025_sequences_pipeline

This is the main feature + sequence build path.

Option 3: Train GRU model

.\.venv\Scripts\python.exe -m src.pipelines.run_gru_propensity_pipeline

Run this after sequence table is available.

5) Vertex AI run (managed cloud execution)

Step 1: Build and push Docker image

gcloud auth configure-docker us-central1-docker.pkg.dev
docker build -t us-central1-docker.pkg.dev/<PROJECT_ID>/<REPO>/cloud-propensity:latest .
docker push us-central1-docker.pkg.dev/<PROJECT_ID>/<REPO>/cloud-propensity:latest

Step 2: Put image URI in .env

Set VERTEX_IMAGE_URI to the pushed image path.

Step 3: Submit Vertex job

.\.venv\Scripts\python.exe -m src.pipelines.run_on_vertex_custom_job --display-name cloud-propensity-policy-2024-2025

Run GRU pipeline on Vertex

.\.venv\Scripts\python.exe -m src.pipelines.run_on_vertex_custom_job --pipeline gru_propensity --display-name cloud-propensity-gru

6) Important safety settings

  • PIPELINE_RECREATE_BQ_TABLES=true: drops/recreates tables before load (strong reset behavior).
  • PIPELINE_RECREATE_BQ_TABLES=false: safer when runtime account cannot delete tables (common on Vertex).
  • VERTEX_STAGING_BUCKET must be in same region as GCP_REGION.

7) Troubleshooting (simple checklist)

  1. Are you using venv? Prompt should show (.venv) or use .\.venv\Scripts\python.exe.
  2. Is GCP_PROJECT_ID set?
  3. Did ADC login complete?
  4. Are the 3 CSV files present in GCS path?
  5. Does your account/service account have BigQuery + GCS permissions?
  6. If Vertex fails, open Vertex logs and use the first real stack trace line.

Common error meanings

  • No module named google.cloud - you ran outside venv.
  • Table with field based partitioning must have a schema - table create/load schema mismatch.
  • No 30-day sequences found - no data in the 30-day pre-purchase window.
  • Vertex worker exited non-zero - cloud container failed; check Vertex logs URL for exact cause.
